# プロジェクトステータス（大分類・確定）

プロジェクト（Transaction）の**大分類**ステータスを定義する。旧来の「受注待ち」と「要因確定待ち」は**同時進行しうる**ため、**1つの大分類**にまとめる。大分類「受注・要因確定待ち」の**小分類**は下記の**案**として列挙する（実装・運用で増減してよい）。大分類「完了待ち」のうち**メンバー管理の観点の小分類**は §7.1 に定義する。

## 1. 目的

- 一覧・レポート・通知の**粗い分類**を共通言語で固定する。
- 各大分類名が**次に何を待っているか**を示す（「～待ち」形式）。

## 2. データモデル上の前提

- プロジェクトは Transaction（Tr）として扱う（`docs/data-model-flowchart.md`）。
- 本アプリでは**営業が提案段階の案件に対して技術者を予約する**ことが中心である。従来型の「誰かが一括でプロジェクトにアサインする」モデルではない。
- 大分類は上記の利用シーンに合わせ、**獲得・商業条件のロック前（予約を含む）→実行完了→事務クローズ**の流れで説明できるようにする。

## 3. 大分類（確定）

大分類どうしの順序はライフサイクルの**目安**であり、厳密な必須順序は業務ルールで別途定義する。

| 順 | 表示名 | いま待っているもの（次の扉） |
|----|--------|------------------------------|
| 1 | 受注・要因確定待ち | 顧客からのコミット（受注・契約など）と、単価・体制・期間など**要因**の合意・確定。両者は**同時進行しうる**ため同一帯にまとめる。 |
| 2 | 完了待ち | 成果・稼働の**完了**（納品・期間満了など）。 |
| 3 | クローズ待ち | 請求・記録・引き継ぎなど**終了処理**の完了。 |

### フローチャート（大分類・小分類）

subgraph のタイトルが**大分類**、内側のノードが**小分類**。受注・要因確定待ちの小分類は**案**であり、**小分類どうしに固定順序はなく**往復しうる（§5）。完了待ちのメンバー管理小分類は §7.1 の定義に従い、**混在時の代表表示の優先ルール**がある。大分類どうしの矢印は本流の目安である。同図は `docs/data-model-flowchart.md` にも掲載する。

```mermaid

  end
```

## 4. 各大分類の役割（補足）

### 4.1 受注・要因確定待ち

- **受注前〜商業条件がビジネス上ロックされるまで**の帯。見積の差し戻し、継続案件からの**要因引き継ぎ直後**などもここに含む。
- 営業による**要員予約**は主にこの大分類で行う想定。開始日が未来でもよい（先取りの予約と両立する）。
- **小分類**は「いま主に詰まっている窓口」を表す。**順序は絶対ではなく**、同一案件で小分類を行き来してよい（例: プロジェクト小分類と**予約の承認フロー**が並走し、往復してよい）。

### 4.2 完了待ち

- **受注および要因がビジネス上確定済み**。**現場の実行から完了まで**を表す。
- **欠員・別人への付け替え、複数プロジェクト間のメンバー交換**などは、原則**大分類は完了待ちのまま**とし、予約・稼働割当の変更や**小分類（§7.1 メンバー管理）**で表す。

### 4.3 クローズ待ち

- 実務は終了したが、**帳務・記録・承認**など事務上の〆が残る場合に用いる。
- 〆が不要な運用では、この大分類を使わず**終了扱い**に寄せる設計もありうる（実装・運用で選択）。

## 5. 受注・要因確定待ちの小分類（案）

以下は**たたき台**。列の増減・名称の変更はチーム・画面に合わせてよい。

| 小分類（案） | いま待っているもの（目安） |
|--------------|---------------------------|
| 要員予約待ち | 案件はあるが、まだ誰も枠に乗せていない（次は予約操作）。 |
| 要因入力・起票待ち | 継続引き継ぎや新規で、システム上の要因が未入力またはドラフト。 |
| 社内確認・承認待ち | 原価・採算・役割などの社内OK待ち。 |
| 契約・発注手続待ち | 原則合意済み。発注書・契約書の締結・受領待ち。 |

**顧客とのやり取り**および**条件の修正・差し戻し**は、プロジェクトの小分類では分けず、**予約**の確定待ち（`docs/reservation-status.md` の**承認待ち**・**差戻し対応待ち**・**契約待ち**など）や、確定済みの**顧客合意待ち**で管理する。

**見込み保留**（動きが止まっている）は、小分類に含めるか、**別フラグ**にするか実装で選択する。

**小分類の遷移:** 上から下への**固定シーケンスは想定しない**。必要に応じ任意の小分類へ移り、**往復**してよい。

## 6. 終了・失注

- **クローズ待ちを経たあと**、システム上は「アーカイブ済み」「参照のみ」など**大分類の外**の表現にするか、小分類で表すかは実装時に決める。
- **失注・中止**を大分類のどこに置くか（受注・要因確定待ちからの打ち切し、完了待ちからの中止など）は、別途フラグや小分類と組み合わせて定義する。

## 7. 完了待ち・クローズ待ちの小分類

### 7.1 完了待ち（メンバー管理の小分類）

プロジェクト単位で**体制が健全か、手当てが必要か**を一覧しやすくするための区分である。**誰が・いつから・何％か**の真実は**稼働割当**（と必要なら**予約**の内包情報）に置く（`docs/work-assignment-status.md`、`docs/reservation-status.md`）。表示名「稼働中」は割当側の大分類名「稼働終了待ち」内の小分類「稼働中」と区別が必要な画面では、接頭語などで補ってよい。

| 小分類 | 説明 |
|--------|------|
| 参画待ち | 予約・割当はあるが、開始日・キックオフ前でまだ稼働に入っていない。**メンバー管理上は通常、問題のない健全な前段**（枠は満たされ、開始を待つだけ）。 |
| 稼働中 | **メンバー管理上、問題がない状態**。少なくとも1名が稼働割当上「稼働終了待ち」相当（小分類は**稼働中**）であり、計画体制に不足・欠員・未完了の増員手当てがない（用語は割当側に合わせる）。 |
| 補充・代替手配待ち | プロジェクトは**進行中**だが、工数増加などで**増員が必要**となった場合、または**離脱・欠員**などがあり、別人の予約・承認・割当作成など**体制の手当てが未完了**の状態。 |

**代表表示（混在時）:** 同一プロジェクト内で、メンバーごとに上記が異なりうる。プロジェクトに**1つだけ**小分類を表示する場合、**補充・代替手配待ちが1つでもあればそれを優先**する（手当てが必要な状態を表面化する）。

**稼働割当との対応（目安）:** 割当が「稼働待ち」で開始前は参画待ち寄り、「稼働終了待ち」で体制にギャップがなければ稼働中、欠員・増員で手当て未完なら補充・代替手配待ち。境界は実装・業務ルールで定義する。

**遷移:** 上表はライフサイクルの固定順序ではない。状態に応じて**往復**しうる。

### 7.2 クローズ待ちの小分類

- **本書では案を確定しない**。必要になったときに定義する。
- 命名は大分類・その他の小分類と同様、「次の行動が読み取りやすい」ことを優先できる。

# プロジェクト（Transaction）ステータス草案

引継ぎ用メモ。内容は設計途中の整理であり、確定仕様ではない。

## 1. 目的とスコープ

- トランザクションデータのうち、**プロジェクト**に対するステータス設計の方向性をまとめる。
- **アサイン管理**（要員配置の充足・固定・変動）を主な観点に含める。
- Linear のような **2階層**（第1層＝粗い分類、第2層＝具体ステータス）を想定したうえで、**第1層を5区分**にする案を中心に記載する。

## 2. データモデル上の前提

- データモデル接続図（`docs/data-model-flowchart.md`）に沿い、**プロジェクトは Transaction（Tr）**。
- 関連しうる Tr には **予約**、**割当** などがある。プロジェクトの健全性は **割当の集合が必須ロールを満たすか** と対応づけやすい。

## 3. 参考にした外部の考え方（Linear など）

- **Issue ワークフロー**: カテゴリ（例: Backlog / Unstarted / Started / Completed / Canceled）の下に、チームごとの **具体ステータス** をぶら下げる。
- **Project**: Issue とは別軸で **プロジェクト用ステータス**（例: Backlog / Planned / In Progress / Completed / Canceled）を持ち、一覧やタイムライン向けの高い粒度としやすい。
- 本ドキュメントでは、プロジェクトを **アサイン管理と進行** の軸で扱うため、Linear の語をそのまま採用する必要はない。

## 4. 第1層を3区分にした場合（検討メモ）

次の3つに絞る案も検討した。

| 第1層 | 含意 |
|-------|------|
| 配置未完・修正中 | 必須要員が埋まっていない、または再配置で枠が不安定 |
| 要員固定・進行中 | 必須ロールが満たされ、当面アサインが固定された実行フェーズ |
| 終わり | プロジェクトとしての実務が終了 |

**利点**: 一覧・通知の軸が明確。遷移グラフが小さい。

**注意**: 「開始前の溜め」「打ち切しと正常完了」「一時停止」の区別は第2層や別属性に逃がす必要がある。欠員時に第1層を「配置未完」へ戻すか、「進行中」のまま第2層だけ `Reassigning` にするかは設計選択が必要。

## 5. 第1層を5区分にする案（現在の中心案）

バックログ・未着手、配置調整、固定して進行、終了、一時停止を **第1層で分ける**。

| 順 | 意味 | 役割の要約 |
|----|------|------------|
| 1 | バックログ・開始前 | プロジェクトはあるが、実行前。優先度や要員枠が未整理のことも含む |
| 2 | 配置未完・修正中 | 必須ロールが未充足、または差し替え・再アサインが必要 |
| 3 | 要員固定・進行中 | 必須配置が満たされ、実行フェーズ |
| 4 | 終了 | 完了処理・クローズに向かう／完了した状態（詳細は第2層で分ける想定） |
| 5 | 一時停止 | 再開判断が必要。他区分と直交しやすいため第1層で独立 |

一時停止は **boolean や第2層** に寄せる設計もあるが、本草案では **第1層の1区分** として扱う案を採用している。

## 6. 命名方針

### 6.1 社内用語との関係

- 社内で「キックオフ＝プロジェクト開始」としている場合、**UI・ドキュメントの第1層名では「キックオフ」は使わない**。
- 代わりに **開始前 / 未着手 / 着手前 / 待機** など、社外用語と混同しにくい表現を使う。

### 6.2 過去分詞系（次の行動が読み取りやすい名前）

- **英語の列挙子**は `Paused` / `Staffed` / `Recorded` のように **過去分詞・分詞形容詞系**で揃えると、「ここまで完了している」ことが分かり、**次に足りない作業**を逆算しやすい。
- **日本語UI**は **起票済み / 割当済み / 中断済み** のように **済み・未・保留** を揃えると対応しやすい。

## 7. 候補名（英語スラッグと日本語の例）

確定ではない。実装時は列挙子と表示名を分ける。

| 第1層の意味 | 英語スラッグ例 | 日本語表示例 | 示唆される次の行動（例） |
|-------------|----------------|--------------|---------------------------|
| 開始前・溜め | `Recorded`, `Queued` | 起票済み、登録済み | 優先付け、要員枠・要件の確定 |
| 配置未完・修正中 | `Unassigned` ほか | 未割当あり、割当要修正 | 割当・再割当・承認 |
| 要員固定・進行中 | `Staffed`, `Assigned` | 割当済み、配置確定済み | 実行・進捗管理 |
| 終了 | `Delivered` と `Closed` を第2層で分ける案あり | 納品済み、クローズ済み | 請求・記録・アーカイブ |
| 一時停止 | `Paused`, `Suspended` | 中断済み、保留済み | 再開または打ち切しの判断 |

英語で厳密な過去分詞にこだわる場合、`Unassigned` のように形容詞的な語もワークフロー名としては一般的。避けるなら `AssignmentIncomplete` など複合語も選択肢。

## 8. 第2層（具体ステータス）の役割（たたき台）

第1層の下に、ボード列・運用の細かさを載せる。

| 第1層 | 第2層の例（過去のやりとりより） |
|-------|----------------------------------|
| 開始前・溜め | Draft, RequirementsPending, ScopeFrozen |
| 配置未完・修正中 | Sourcing, PartiallyAssigned, Reassigning |
| 要員固定・進行中 | InExecution, Scheduled（開始待ち） |
| 終了 | Handover, DeStaffing, Settled |
| 一時停止 | OnHold（第1層と名前が被る場合は第2層で細分化） |

必須ロール表と割当 Tr を突き合わせ、**PartiallyAssigned と FullyStaffed（第2層）** から第1層の「配置未完」と「固定・進行」の切り替えを自動化する案が扱いやすい。

## 9. 主な遷移イメージ（草案）

- 開始前から配置調整へ: 要員要件や枠が定義されたとき。
- 配置調整から固定・進行へ: 必須ロールが充足し、開始条件（日付・予約確定など）を満たしたとき。
- 固定・進行から配置調整へ: 欠員・辞退などで再アサインが必要になったとき（第1層を戻すか、第2層のみ Reassigning にするかは未決）。
- 固定・進行から終了へ: 成果・作業の完了条件を満たしたとき。
- いずれからも一時停止へ: 運用ルールで許可範囲を決める。
- 一時停止から: 直前の第1層へ復帰、または打ち切しで終了へ。
- 打ち切し: 原則どの段階からも可能にするか、配置調整までに限定するかは未決。

## 10. 未決事項・次に決めたいこと

- 「要員固定」の **機械的な定義**（全ロール確定のみか、開始日到来か、予約 Tr の状態か）。
- 欠員時に **第1層を配置未完へ戻すか**、**進行中のまま第2層で表すか**。
- 終了を **1つの第1層**にまとめるか、**納品済みとクローズ済み**を第1層で分けるか。
- 打ち切しを第1層「終了」の第2層で表すか、別フラグ（Canceled）にするか。
- 予約・割当 Tr のイベントと、どの遷移を **自動** に結びつけるか。
